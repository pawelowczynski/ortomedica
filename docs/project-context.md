# Orthomedica Project Context (Astro)

## 1) Project Overview

- Marketing/lead website for a dental/orthodontic clinic in Lubin (`https://www.orthomedica.lubin.pl`).
- Static Astro site with mostly server-rendered pages and one shared client script for UI behaviors.
- Main project code lives in [`/`](../), documentation is in [`docs/`](./).

## 2) Tech Stack

- Framework: Astro v6 (`"astro": "^6.0.3"`) in [`package.json`](../package.json).
- Styling: Tailwind CSS v4 via Vite plugin (`@tailwindcss/vite`) in [`astro.config.mjs`](../astro.config.mjs).
- Integrations: `@astrojs/sitemap` in [`astro.config.mjs`](../astro.config.mjs).
- Icons: Lucide loaded from CDN in [`src/layouts/layout.astro`](../src/layouts/layout.astro).
- Runtime requirement: Node `>=22.12.0` in [`package.json`](../package.json).

## 3) Repo & Directory Map

- Repo root: [`/`](../)
- Core app paths:
  - Pages/routes: [`src/pages`](../src/pages)
  - Layout: [`src/layouts/layout.astro`](../src/layouts/layout.astro)
  - Components: [`src/components`](../src/components)
  - Content markdown: [`src/content/blog`](../src/content/blog)
  - Global styles: [`src/styles/global.css`](../src/styles/global.css)
  - Shared client JS: [`public/main.js`](../public/main.js)
  - Build output: `dist` (generated by `npm run build`)

Defined in [`package.json`](../package.json):

- `npm run dev` -> `astro dev`

## 5) Content Architecture

- Blog content is authored as markdown files in [`src/content/blog`](../src/content/blog).
- Blog listing page ([`src/pages/blog/index.astro`](../src/pages/blog/index.astro)):
  - Uses `import.meta.glob('../../content/blog/*.md', { eager: true })`
  - Sorts posts by `frontmatter.date`
- Dynamic blog route ([`src/pages/blog/[slug].astro`](../src/pages/blog/%5Bslug%5D.astro)):
  - Generates static paths from markdown filenames
  - Renders markdown via `const { Content } = await currentPost`
- Main homepage composition is in [`src/pages/index.astro`](../src/pages/index.astro) and stitches multiple Astro components.

## 7) Performance Baseline (Mobile)

Source snapshots:
- User-provided Lighthouse mobile screenshot (Apr 30, 2026).
- PSI run: <https://pagespeed.web.dev/analysis/https-www-orthomedica-lubin-pl/9b56vk143s?form_factor=mobile>

Observed key lab metrics (mobile):

- Performance score: **52**
- Accessibility: **87**
- Best Practices: **100**
- SEO: **100**
- FCP: **3.8s**
- LCP: **8.5s**
- TBT: **470ms**
- Speed Index: **5.6s**
- CLS: **0**

Interpretation:

- Main bottleneck is **LCP** (hero render is too late).
- **FCP** and **TBT** indicate slow first paint + main-thread pressure.
- **CLS** is healthy (layout is stable).
- PSI also reports no CrUX field data for that URL/sample, so optimization should be driven by lab profiling + repeatable tests.

## 8) Performance Playbook (Astro-First)

Priority order (largest expected impact first):

1. LCP hero path
   - Optimize/convert hero background image (`public/images/hero-bg.jpg`) to modern formats and right sizes.
   - Reduce expensive above-the-fold visual effects where possible.
   - Consider replacing CSS background LCP candidate with an explicit `<img>`/`<picture>` if it improves preloadability.

2. Critical rendering path
   - Reduce render-blocking font cost in [`src/layouts/layout.astro`](../src/layouts/layout.astro) (Google Fonts request chain).
   - Evaluate self-hosting and tighter font subsets/weights.

3. Main-thread JS pressure
   - Audit [`public/main.js`](../public/main.js) for non-critical logic running on initial load.
   - Split deferred behaviors (map/forms/observers/sliders) by route/visibility where feasible.
   - Replace global `DOMContentLoaded` bundle behavior with smaller feature-specific modules when practical.

4. Third-party cost
   - Keep analytics and map loading behind consent (already implemented).
   - Verify Lucide CDN cost and consider static/partial icon strategy if payload impact is high.

5. Repeatable performance loop
   - Run mobile Lighthouse after each significant change.
   - Track trend for LCP/FCP/TBT in this document’s decision log.

## 9) Operational Notes

Quick post-change checklist:

- `npm ci` (if lockfile/deps changed)
- `npm run build` succeeds
- `npm run preview` smoke test critical routes (`/`, `/blog`, representative `/zabiegi/*`)
- Re-run Lighthouse mobile and compare against baseline in section 7
- Confirm no regressions in cookies banner, navigation/menu, contact form, and map lazy-load

## 10) Glossary & Decisions Log

Glossary:

- **LCP**: Largest Contentful Paint, primary metric for perceived load speed.
- **TBT**: Total Blocking Time, proxy for responsiveness during load.
- **CrUX**: Real-user Chrome dataset used by PSI for field data.
